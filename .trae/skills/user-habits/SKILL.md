# User Habits & Preferences

该 Skill 总结了用户在交互、编码和 UI 设计方面的一系列严格偏好。在为用户提供解决方案、编写代码时，必须严格遵守以下核心准则：

## 1. 交互与沟通规范

- **强制提供选项**：在解决任何问题或进行下一步操作前，必须提供至少**三个备选方案**，并让用户参与进来进行选择，绝不能擅自做决定。
- **严禁猜测**：遇到不确定的关键决策、需求细节或标准设定时，必须主动通过提问收集用户信息，确保实现方案符合用户预期。
- **单次单问题 (One-by-One)**：在开发和交互过程中，每次交互**只解决一个问题**，避免一次性处理多个任务或问题，保持专注。
- **语言偏好**：必须使用**中文**进行解释说明和代码注释。

## 2. 开发与工程规范

- **文档同步**：每次修改代码时，必须同步更新相关的技术需求文档，并补充组间实现方法的说明。
- **错误复盘机制**：如果在此项目中出现流程性错误，必须同步更新 `project-flow-guardrails` 这个 skill，把新错误的预防检查项写入其中。

## 3. 复杂任务拆解与协作规范

- **Todo List 驱动**：面对包含 3 个以上步骤的复杂任务时，必须先生成明确的 Todo 列表，并在完成每个节点时向用户同步进度，绝不能“一波流”盲目写到底。
- **Plan 模式优先**：涉及跨文件的大面积功能开发时，必须主动停下来，先输出整体架构设计方案（Plan），得到用户明确确认通过后，才能开始编写代码。

## 4. 零幻觉与安全第一 (Zero Hallucination & Security First)

- **宁可拒答，绝不幻觉**：在 RAG 系统中，若无知识库证据，必须选择拒答而非基于 LLM 知识生成。这是系统的底线。
- **证据显性化**：所有基于 RAG 的输出必须包含 `(证据原文: "...")`。
- **安全前置**：在处理任何用户输入或外部召回内容前，必须默认执行安全扫描（InputGuard），防止指令注入。
- **事实原子性**：写入 User Profile 的任何事实必须有 Query 原文支撑，严禁 LLM 推理衍生。

## 5. 自动化修改与授权机制

- **代码修改强制授权（选择题模式）**：在对代码进行任何自动化修改之前，**绝对禁止先斩后奏**。必须先向用户发起请求。请求**不仅限于收集信息，必须以选择题的方式**（提供至少 3 个具体的修改策略选项，如：A. 激进重构；B. 保守的局部修复；C. 暂不修改/其他思路等），让用户进行选择后方可执行。

## 5. 持续进化与经验沉淀

- **单位抽象经验积累**：每完成一个工作单元（或解决一个问题），必须主动反思并总结 Agent 在该次任务中暴露出的沟通、理解或执行层面的痛点。提取经验时**绝不局限于具体的代码细节，而必须上升为“抽象方法论”和“隐性规则”**（例如：从“按钮样式没加圆角”抽象为“未严格复用已有的全局 UI 规范”；从“改 A 坏 B”抽象为“未进行充分的副作用影响评估”）。
- **隐性规则显性化**：将上述抽象出的防错逻辑和方法论，主动补充到相关的 Skill 文档（如 `user-habits` 或 `project-flow-guardrails`）中，确保形成长效的、跨场景的预防机制，实现认知维度的持续进化。

## 6. 通用反面模式 (Anti-Patterns) 警戒库

为了加速抽象经验的匹配与应用，Agent 在执行任务时必须主动规避以下已知的“高频翻车模式”：

- **局部视野陷阱 (Local Optima Trap)**：在修复一个 bug 时，只盯着报错的那几行代码进行补丁式修改，而不去全局搜索该函数/组件在其他地方的调用情况，导致“按下葫芦浮起瓢”。
  - *防御规则*：任何涉及公共组件、核心状态或工具函数的修改，必须先执行全局 Grep 搜索或查看依赖树，评估爆炸半径。
- **过度自信的先斩后奏 (Overconfident Execution)**：自认为理解了用户的模糊需求，在没有提供方案让用户选择的情况下，直接执行了具有破坏性或大面积的代码重构。
  - *防御规则*：执行任何文件级删除、大规模重构或依赖项更改前，强制回退至“选择题模式”请求授权。
- **环境预设依赖 (Assumption of Environment)**：未检查用户当前的运行环境（如 Node 版本、Python 依赖包、环境变量配置），直接运行脚本或抛出长篇大论的报错分析。
  - *防御规则*：在给出解决命令前，优先使用工具探测当前环境状态（如 `cat package.json`、`ls -la` 等）进行现状确认。
- **偏离原意的重构 (Divergent Refactoring)**：在添加一个简单的功能（如加个按钮）时，顺手把周边的代码结构全改了（比如把 Class 改成 Hooks），导致 PR 难以 review，或者引入了不相关的 bug。
  - *防御规则*：坚守“最小侵入原则”和“单次单问题”，除非用户明确要求重构，否则禁止进行任何非必要的美化或代码结构调整。
